## 核心立场

这不是“多个 persona 一起聊天”。
这是一个 **多法庭席位 / 多失败入口** 的批判性审议系统。

## 1. Review Seats

### 职能席位版
- 产品席
- 设计席
- 技术席
- 用户现实席
- 领导 / 业务席
- 品牌 / 定位席
- 反方席
- 主审席

### 失败模式版
- `value_failure`
- `comprehension_failure`
- `execution_failure`
- `adoption_failure`
- `strategic_failure`
- `narrative_failure`
- `reasoning_failure`

建议：
默认以职能席位表达，对深度评议再映射到失败模式。

## 2. Seat Responsibilities

### Product Seat
问：
- 这到底在解决什么核心问题？
- 是长期任务还是一次性炫技？
- 新变化是局部增强还是方向转向？

### Design Seat
问：
- 用户是否知道系统当前状态？
- 什么时候该信、该介入、该纠错？
- 是否更智能但更难理解？

### Technical Seat
问：
- 这是结构能力还是 prompt 拼装？
- 上下文、成本、延迟、稳定性是否支撑承诺？
- 失败恢复机制在哪里？

### User Reality Seat
问：
- 用户真的会用吗？
- adoption friction 在哪里？
- 用户抱怨是显性问题还是更深心智断裂？

### Leadership / Business Seat
问：
- 值得进优先级吗？
- 需要什么组织前提？
- 规模化后的 ROI 是否成立？

### Brand / Positioning Seat
问：
- 这件事是否强化“我们是谁”？
- narrative 与内部机制是否一致？
- 是 category 武器还是功能碎片？

### Contrarian Seat
职责：只负责系统性找错。
攻击点：
- 偷换概念
- 证据不足
- 把营销话术当机制事实
- 把 demo 当稳定能力
- 把某类用户价值误当普遍价值

### Synthesis Judge
只负责裁定：
- 哪些结论证据足够
- 哪些必须降级表达
- 哪些冲突应保留，不强行收束

## 3. Review Modes

### Mode A｜产品策略型
权重偏：产品 / 业务 / 品牌 / 用户现实

### Mode B｜系统机制型
权重偏：技术 / 设计 / 产品 / 反方

### Mode C｜落地可行型
权重偏：技术 / 业务 / 用户现实

### Mode D｜信任与体验型
权重偏：设计 / 用户现实 / 品牌 / 产品

## 4. Review Protocol

### Phase 1｜案件立案
- 分析对象是谁
- 本轮时间窗口是什么
- 本轮要回答的问题是什么
- 哪些 source 可用 / 不可用

### Phase 2｜证据卷宗
由 Source Scout / Evidence Clerk 汇总 source。
不允许先下结论。

### Phase 3｜独立初审
每个席位独立提交：
- 最看重的 3 个结论
- 最危险的 3 个问题
- 最不相信的 2 个说法
- 还缺什么证据

### Phase 4｜交叉质询
席位之间互相质疑，不做礼貌补充。

### Phase 5｜修正版立场
被质询方必须：
- 修正判断，或
- 明确坚持原判断，并说明理由

### Phase 6｜主审裁定
输出三类结果：
- 成立判断
- 待证判断
- 明确保留的分歧

## 5. Output Shape

最终输出至少四层：
- 事实层
- 机制层
- 用户层
- 判断层

同时保留：
- 争论过程
- 被推翻的早期判断
- 长期冲突点
- 最容易误导你的 source 类型
